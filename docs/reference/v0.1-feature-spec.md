# WorkNexus v0.1 功能规格书（工作底稿）

> 定位：各模块开发时，从本文件对应章节展开成 `docs/modules/<module>.md` 的底稿。
> 效力低于 `AGENTS.md`（规范）与 `docs/roadmap.md`（决策 D1–D10）；冲突时以后两者为准。
> 所有 API 路径统一带版本前缀 `/api/v1`；MCP 端点为 `/mcp`。

---

## 0. v0.1 核心判断

v0.1 不是"带 AI 的工单系统"，而是**能让 AI 安全参与企业工作流的协作底座**。优先保证：权限链路正确、AI 身份链路正确、业务数据归 WorkNexus 所有、multirag 只负责智能能力、AI 写动作必须确认和审计。

通用表字段约定：所有核心业务表必有 `id / tenant_id / created_at / updated_at`；重要业务表另加 `created_by / updated_by`（软删除 `deleted_at` 按需）。

## 1. Identity & Access（M1，后端 module: identity）

### 数据表

`tenants`、`users`、`sessions`、`invite_tokens`、`project_members`、`role_bindings`、`ai_agents`、`mcp_delegation_tokens`、`audit_logs`（横切起步）

- users 关键字段：`email`、`display_name`、`avatar_url`、`password_hash`（bcrypt，可空）、`identity_provider(local|multirag|oidc)`、`external_user_id`、`status(active|invited|disabled)`、`last_login_at`
- sessions：只存 `token_hash`，含 `ip_address`、`user_agent`、`expires_at`、`revoked_at`、`last_seen_at`
- mcp_delegation_tokens：`token_hash`、`user_id`、`agent_id`、`project_id`、`conversation_id`、`run_id`、`permissions_snapshot`（签发时权限快照）、`expires_at`（TTL 5–10 分钟）、`revoked_at`

### Setup 首启流程

`/setup`：检查无 owner → 创建 default tenant（Default Workspace）→ 首个 owner 用户 → 默认项目（WorkNexus Internal）→ 默认 AI Agent（WorkNexus Assistant）→ 初始化系统角色常量 → 封禁 setup。

### 邀请流程（v0.1 不接邮件）

owner/admin 创建邀请 → 生成 invite token → 复制链接 → 被邀请人设密码激活 → 写入 project_members 或 role_bindings。

### API

```text
GET  /api/v1/setup/status        POST /api/v1/setup
POST /api/v1/auth/login          POST /api/v1/auth/logout
GET  /api/v1/me
GET  /api/v1/users
POST /api/v1/invites             GET  /api/v1/invites/{token}
POST /api/v1/invites/{token}/accept
```

### CurrentUserContext（/api/v1/me 返回）

`user{id,email,displayName,avatarUrl,identityProvider,externalUserId}`、`tenant{id,name,slug}`、`roles[]`、`permissions[]`、`projects[{id,name,role,permissions}]`、`ai{availableAgents[]}`。前端据此控制菜单/按钮/AI 确认入口；后端永远强制校验。

### 验收标准

首启可建 owner；登录写 HttpOnly Cookie；/me 返回完整权限上下文；未登录不能访问业务 API；project_members 控制项目访问；role_bindings 可给 AI Agent 授权；登录/邀请/权限变更全部写 audit log。

## 2. Projects（M2）

- 字段：`name`、`key`、`description`、`status(active|archived)`、`owner_id`、`settings`
- 成员角色：`project_admin | member | viewer`（存 project_members，禁止与 role_bindings 双写）
- 概览展示：名称/描述/负责人/成员数/工作项数/高优先级数/逾期数/AI 创建数/最近活动

```text
GET|POST  /api/v1/projects
GET|PATCH /api/v1/projects/{id}        POST /api/v1/projects/{id}/archive
GET|POST  /api/v1/projects/{id}/members
PATCH|DELETE /api/v1/projects/{id}/members/{user_id}
```

权限点：`project.read/create/update/archive/member.manage`。
验收：项目列表只返回当前用户有权限的项目；viewer 只读；变更写 audit log。

## 3. Work Items + Workflow Lite + Board（M3，module: work_items）

### 类型（8 种）

`task / requirement / bug / risk / decision / approval / incident / feedback`（前端：任务/需求/缺陷/风险/决策/审批/故障/反馈）

### 字段

基础：`key`、`type`、`title`、`description`、`status`、`priority(low|medium|high|urgent)`、`assignee_id`、`reporter_id`、`due_at`、`tags`、`source(manual|ai_chat|intake|mcp|api)`、`source_ref_id`、`ai_summary`、`acceptance_criteria`、`custom_fields(JSONB)`、`created_by`、`updated_by`

类型专属字段 v0.1 全部放 `custom_fields`：

- Bug：severity、steps_to_reproduce、expected_result、actual_result、environment、affected_version
- Requirement：business_goal、user_value、acceptance_criteria、boundary_conditions、dependencies
- Risk：risk_level、impact、probability、mitigation_plan、trigger_condition
- Decision：background、options、decision_result、decision_owner、impact_scope
- Approval：approval_type、approvers、approval_status、approval_comment

### 关系（work_item_relations）

`parent_child / blocks / blocked_by / duplicates / relates_to / created_from_message / created_from_intake`。v0.1 只做关系表与展示，不做关系图。

### Workflow Lite（固定状态机）

状态：`backlog → todo → in_progress → review → done`；`review → in_progress` 允许回退；任意非 done → `cancelled`。状态流转写活动日志 + audit log；AI 可建议下一状态但不能绕过确认。

### 评论与活动日志

评论 Markdown、来源 `user/ai_agent/system`；活动记录创建/改标题/改描述/改负责人/改优先级/状态流转/评论/AI 创建/Intake 转化。

### Board

列 = 6 状态；卡片展示负责人/优先级/类型/AI 来源标记。拖拽有现成能力就做，否则 v0.1 先用状态按钮保稳定。

### API

```text
GET|POST   /api/v1/projects/{project_id}/work-items
GET|PATCH|DELETE /api/v1/work-items/{id}
POST       /api/v1/work-items/{id}/transition
GET|POST   /api/v1/work-items/{id}/comments
GET        /api/v1/work-items/{id}/activities
POST       /api/v1/work-items/{id}/relations
DELETE     /api/v1/work-items/{id}/relations/{relation_id}
```

权限点：`work_item.read/create/update/delete/transition/comment/assign`。
表：`work_items`、`work_item_comments`、`work_item_activities`、`work_item_relations`。
MCP tools（本模块 `mcp.py`）：`create/update/search/get/comment/transition_work_item`（写动作 low_write tag）。

验收：8 类型可建；列表+看板可用；流转写双日志；详情见评论/活动/来源/AI 摘要；AI 创建的有 source 标记；权限不足不可见不可改。

## 4. Skills / MCP 骨架（M4，module: skills）

### MCP 双 token 认证（先于 WorkChat 落地）

每个 MCP 请求校验：① multirag server token（确认请求来自 AI 平台）② `X-WorkNexus-Delegation` delegation token（确认 AI 代表哪个用户/tenant/project/conversation/run）。**禁止用 tool 参数传 user_id 作为身份依据。**

### v0.1 Skill 清单

| Skill | Tools |
| --- | --- |
| workitem-skill | create / update / search / get / comment / transition_work_item |
| project-skill | get_project_context、list_user_projects、get_project_summary |
| intake-skill | create_intake_request、accept_intake_request、reject_intake_request |
| report-skill | get_project_summary、get_overdue_items、get_status_distribution、get_user_workload |
| knowledge-skill | search_knowledge（代理 multirag，不在 WorkNexus 重做知识库） |

### skill_invocations（全量留痕）

`skill_code`、`tool_name`、`caller_type`、`caller_id`、`represented_user_id`、`agent_id`、`project_id`、`input_summary`、`output_summary`（不存敏感完整明文）、`status`、`risk_level`、`requires_confirmation`、`agent_action_id`、`error_message`、`finished_at`

### API / 页面

```text
POST /mcp
GET /api/v1/skills
GET /api/v1/skills/invocations          GET /api/v1/skills/invocations/{id}
```

页面 `/skills`：Skill 列表、Tool 列表、风险等级、最近/失败调用、调用方、代表用户、是否产生 AgentAction。

验收：multirag 可调 /mcp；双 token 校验生效；能识别 represented_user；调用前后有 skill_invocation；权限不足被拒；tool 参数无法伪造身份。

## 5. WorkChat + multirag Adapter + AgentAction（M5，module: workchat）

### 职责分工

WorkNexus：构建用户可见上下文（**先过权限过滤**）、生成 delegation token、调 multirag（SSE）、展示回复、解析 proposed_action、生成 AgentAction。
multirag：模型推理、知识库检索、Agent 编排、MCP 工具规划。

### 调用

`POST {multirag}/api/v1/agents/{agent_id}/completions`，`Authorization: Bearer <APIToken>`，SSE，`custom_header` 携带 delegation token。

### Conversation / Message

每项目一个默认 AI conversation；消息类型 user/ai/system；消息可关联 AgentAction 与 WorkItem。

### AI 返回三类

```ts
type AIResponse =
  | { type: 'message'; content: string }
  | { type: 'proposed_action'; actionType: string; payload: any; requiresConfirmation: boolean }
  | { type: 'knowledge_result'; content: string; references: any[] }
```

### AgentAction

状态：`pending / approved / rejected / executed / failed / expired`。
v0.1 支持动作：`create_work_item`、`update_work_item`、`transition_work_item`、`comment_work_item`、`create_intake_request`、`accept_intake_request`、`search_work_items`、`get_project_summary`（写动作默认确认）。
执行链路：proposed_action → 建 agent_action → 确认卡片 → 用户确认 → 后端双重校验（用户 ∧ Agent ∧ 资源 ∧ 风险 ∧ 确认）→ 业务写入 → skill_invocation → audit_log → 更新状态。

### API / 页面

```text
GET  /api/v1/projects/{project_id}/conversations
GET|POST /api/v1/conversations/{conversation_id}/messages
POST /api/v1/workchat/runs              GET /api/v1/workchat/runs/{run_id}
GET  /api/v1/agent-actions              GET /api/v1/agent-actions/{id}
POST /api/v1/agent-actions/{id}/approve POST /api/v1/agent-actions/{id}/reject
```

页面 `/projects/{id}/ai`：消息列表、输入框、SSE 流式、AI 建议动作卡片（确认/拒绝）、关联工作项入口、知识引用卡片。
表：`conversations`、`messages`、`agent_actions`。

验收：真实调通 multirag 流式；proposed_action 不直接落库；high_write 被禁止执行；AgentAction 全链路可追溯 requested_by/agent_id/approved_by/executed_at；失败有错误记录；AI 上下文只含当前用户可见数据。

## 6. Intake（M6）

- 来源：`manual / ai_chat / api / mcp`（email、IM 渠道 v0.4+，字段预留）
- 状态：`new / triaging / accepted / rejected / duplicate / snoozed / converted`
- 字段：`title`、`description`、`source`、`source_ref_id`、`status`、`submitter_id`、`ai_summary`、`ai_category`、`suggested_type`、`suggested_priority`、`suggested_assignee_id`、`duplicate_work_item_id`、`converted_work_item_id`

```text
GET|POST /api/v1/projects/{project_id}/intake
GET|PATCH /api/v1/intake/{id}
POST /api/v1/intake/{id}/accept|reject|mark-duplicate|snooze
```

权限点：`intake.read/create/triage/accept/reject`。页面 `/projects/{id}/intake`：列表、状态/来源筛选、AI 推荐信息、操作按钮。

验收：WorkChat 可创建 Intake；接受生成 WorkItem 且保留来源关系（created_from_intake）；拒绝/重复写双日志；列表可按状态来源过滤。

## 7. Dashboard（M7）

指标：工作项总数、按状态/类型/优先级/负责人统计、逾期数、高优先级数、AI 创建数、Intake 请求与转化数、近 7 天创建/完成趋势。
AI 洞察：风险摘要、逾期摘要、高优先级摘要、负载摘要——**可调 multirag 或先由规则生成**（v0.1 允许规则版起步）。

```text
GET /api/v1/projects/{project_id}/dashboard/summary
GET /api/v1/projects/{project_id}/dashboard/workload
GET /api/v1/projects/{project_id}/dashboard/overdue
GET /api/v1/projects/{project_id}/dashboard/ai-insights
```

验收：统计来自真实数据并随状态变化更新；AI 创建数可统计；逾期可查看。

## 8. Audit（横切 + M8 UI）

必记动作：setup 建 owner、登录/登出、邀请/接受邀请、项目增改归档、成员变更、工作项增删改/流转、评论、Intake 接受/拒绝/重复、AI proposed_action 创建、AgentAction 确认/拒绝/执行、MCP tool 调用、权限绑定变更、Skill 调用失败、设置变更。

AI 动作可追溯链：user → agent → action → approval → execution（requested_by / agent_id / approved_by / skill_invocation_id）。

```text
GET /api/v1/audit-logs
GET /api/v1/projects/{project_id}/audit-logs
```

页面 `/audit` 全局列表 + 项目筛选；支持按 actor/resource/project/action 过滤；权限点 `audit.read`。

## 9. Settings Lite（M8）

- 用户：个人资料、退出登录
- 项目：名称/描述/成员/归档
- 工作项：类型/状态/优先级展示（v0.1 固定，不做配置）
- AI 连接：multirag endpoint、APIToken、默认 agent_id、MCP server token —— **v0.1 以 .env 配置为准，设置页只读脱敏展示**；落库可配后置
- Skills：启用/禁用基础 skill、查看 tool 风险等级

页面：`/settings`（profile/security/ai）、`/projects/{id}/settings`。
验收：member 不能改系统 AI 配置；敏感 token 不明文展示；设置变更写 audit log。

## 12. v0.1 总验收链路（16 步）

```text
1 首启 /setup 创建 owner          9  WorkChat 调用 multirag
2 owner 登录                     10 AI 返回创建工作项建议
3 owner 创建项目                  11 WorkNexus 生成 AgentAction
4 owner 邀请 member               12 用户确认
5 member 登录后只见自己项目        13 WorkNexus 创建工作项
6 member 创建工作项               14 skill_invocations 记录调用
7 member 在看板流转状态           15 audit_logs 记录建议/确认/执行
8 member 打开 AI WorkChat        16 dashboard 统计更新
```
